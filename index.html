Prompting

Briefing một cộng sự mới: công thức C-T-R

Prompt không phải ra lệnh — prompt là brief. Càng rõ — càng đúng.

C — Context

Bối cảnh

Bạn là ai, đối tượng người đọc là ai, bối cảnh hiện tại của dự án là gì.

"Tôi là Account Lead Kaopiz, khách là CTO startup Nhật 50 người, lần đầu hợp tác."
T — Task

Nhiệm vụ

Động từ hành động cụ thể rõ ràng (Viết, Phân tích, Tổng hợp, Đánh giá).

"Viết 3 phương án email cảm ơn khách đã ký HĐ, có CTA đặt kickoff call."
R — Rules

Ràng buộc

Tone, định dạng (bullet, bảng), độ dài, và những gì không được làm.

"Tiếng Nhật 丁寧語, < 180 từ, không emoji, ký tên Sales Lead Nguyễn A."

Mẹo Pro: Đừng dùng Claude để viết code/email ngay. Hãy dùng nó để hỏi về codebase / context trước. Một prompt cụ thể 50 từ tiết kiệm 20K token Claude phải đoán mò.

💡 Ví dụ thực — C-T-R cho 2 lĩnh vực khác nhau

Marketing

C: "Tôi là Content Lead Kaopiz, audience SME tech VN, brand voice tối giản."
T: "Viết 3 caption Instagram launch service AI Code Review."
R: "≤ 150 ký tự, hook contrarian, kết bằng câu hỏi, không emoji loè loẹt, tag @kaopizvn."

Dev

C: "Project React 18 + TS strict, ESLint Airbnb, hooks-only. calculateTax hiện viết JS thuần, có 4 caller."
T: "Refactor sang TS strict, giữ signature, return type là TaxResult."
R: "Không sửa caller; thêm 3 unit test (happy + 2 edge); không dùng any."

📝
Description

Mô tả bằng C-T-R. Output sai là tín hiệu để sửa prompt, không phải lỗi của AI.

🤝
Delegation

Chọn đúng việc để giao cho AI — thay vì tự làm. Việc lặp lại, nhàm chán, có template? Giao.

🔍

Lightweight Evals — xây niềm tin có cơ sở

1
Chuẩn bị 'Golden Set'

Lấy 1 task bạn đã tự làm xong hoàn hảo làm "đáp án chuẩn"

2
Chạy test

Tạo test prompt + upload file context cho Claude chạy thử

3

Đối chiếu output của Claude với đáp án (Accuracy, Structure, Voice)

Rule of thumb: Sau 3 lần iteration vẫn không ưng ý → prompt gốc sai tư duy, hãy viết lại từ đầu.

💡 Ví dụ thực — 4D cho task "Viết blog post 800 từ về Claude Skills"

  • 📝 Description: CTR đầy đủ — "Bạn là DevRel Kaopiz, audience dev SME VN. Viết blog 800 từ giới thiệu Claude Skills, hook contrarian. Format: H2 + bullet, có 1 code-block SKILL.md mẫu, kết bằng CTA dùng thử."
  • 🤝 Delegation: Giao Claude phần outline + draft + 3 ý tagline. Bạn giữ lại phần kể câu chuyện thực tế Kaopiz đã làm (Claude không biết).
  • 🔍 Diligence: Verify số liệu Anthropic (token tiết kiệm, % team dùng) — chỉ trích từ docs chính thức, không tin Claude tự bịa.
  • ⚖️ Discernment: Check Accuracy (số liệu khớp), Structure (H2 rõ, không sa đà), Voice (giọng Kaopiz — tối giản, không hype). Iteration 2-3 vòng trước publish.
Prompting

6 nguyên tắc + 8 lỗi cần né

6 nguyên tắc vàng

  • Cụ thể, đo được — "250 từ, hook contrarian" thay vì "hay hay"
  • Đưa ví dụ — 2-3 ví dụ output mong muốn (few-shot)
  • XML tag chia khối — <context>, <task>, <example>
  • Nghĩ từng bước — task suy luận tăng accuracy
  • Lặp & tinh chỉnh — "ngắn hơn 30%" — đừng làm lại từ đầu
  • Chỉ định format — JSON? Markdown? 3 phương án?

8 lỗi phổ biến

  • Prompt 1 dòng — quá ít context
  • Hỏi xong copy luôn — không tinh chỉnh
  • Một chat đa task — Claude lẫn context
  • Không feedback — sai mà không bảo "sai chỗ X"
  • Tin tuyệt đối — số liệu vẫn cần verify
  • Không upload tài liệu nguồn — bắt Claude đoán
  • Sợ prompt dài — 500 từ tốt hơn 5 từ
  • Không lưu lại — mỗi lần làm lại từ đầu

Mẹo Pro: Sau mỗi prompt thành công, dán vào file my-prompts.md. 3 tháng sau bạn có "kho prompt" cá nhân. Khi đủ thuần, đóng gói thành Skill (xem phần Skills).

❌ vs ✅ — Cùng 1 task "Viết caption LinkedIn"

❌ SAI
viết caption LinkedIn hay hay về AI cho công ty mình

Lý do: 1 dòng, không context (công ty nào, audience ai), không format (mấy phương án), không ràng buộc (độ dài, tone). Output sẽ generic, kiểu chung chung — gửi đi sẽ phải viết lại 80%.

✅ ĐÚNG
Bạn là Marketing Lead Kaopiz (công ty phần mềm VN, brand voice tối giản, hướng dev). Viết 3 phương án caption LinkedIn giới thiệu Claude Skills cho team. Audience: CTO/Eng Manager SME VN. Ràng buộc: - Mỗi caption ≤ 200 từ - Hook contrarian (đảo narrative) - Kết bằng câu hỏi mở - Không emoji loè loẹt - Tag @kaopizvn cuối

Lý do: Có đủ C-T-R. Output 3 phiên bản → chọn 1, sửa 5%. Tiết kiệm 1 giờ vs viết tay.

BÀI 01

Tóm tắt tài liệu dài thành bullet

DỄ ⏱ 5 phút 🛠 Claude.ai · file upload 🎯 Mọi vai trò

Tình huống

Sếp gửi báo cáo PDF 25 trang lúc 9h sáng, hỏi ý kiến lúc 10h. Bạn cần "vắt sữa" tài liệu trong 5 phút.

📥 Input mẫu
File: kaopiz-q1-2026-board-report.pdf (28 trang)
Nội dung: doanh thu, headcount, project milestones, risks

Prerequisite

  • Tài khoản Claude.ai (free OK)
  • File PDF/DOCX/Excel có nội dung thật

Common pitfalls

  • Quên upload file — chỉ paste tên file
  • Hỏi câu mơ hồ "tóm tắt giúp" — output chung chung
  • Không yêu cầu tiếng Việt → Claude trả lời tiếng Anh

Prompt mẫu (copy & chỉnh)

Bạn là analyst đọc tài liệu B2B. Đọc file đính kèm và trả về: 1. TL;DR: 1 câu (≤ 30 từ) 2. 5 điểm chính: bullet, mỗi cái 1-2 câu 3. 3 con số quan trọng 4. Action items: việc cần làm, ai làm, deadline (nếu có) 5. Câu hỏi mở: điều tài liệu chưa rõ, cần verify Tiếng Việt, dùng từ kinh doanh chuyên nghiệp, tránh sáo rỗng.

Success criteria

  • Output ≤ 1 màn hình, đọc xong hiểu ngay
  • Có ít nhất 1 con số trong "3 con số quan trọng"
  • "Câu hỏi mở" giúp bạn ra meeting với câu hỏi thông minh

Variations để tự thực hành

  • Tóm tắt 1 video transcript YouTube
  • Tóm tắt thread Slack 200 messages
  • So sánh 2 file PDF khác nhau
📤 Output mẫu (click mở rộng)
TL;DR: Q1 vượt 12% kế hoạch nhờ AI service; headcount +8, margin giữ 23%; 2 risk cần xử lý trước Q2.

5 điểm chính:
• AI Consulting đóng góp 38% revenue Q1, gấp đôi Q4/2025.
• Java team thêm 8 dev — onboard 14 ngày OK, nhưng 3 dự án chậm 1-2 tuần.
• Khách Nhật chiếm 62% pipeline mới, US 28%, EU 10%.
• Margin giữ 23% nhờ giảm cost văn phòng (mở rộng WFH).
• CSAT 4.6/5, NPS +18 so Q4.

3 con số: Revenue +12% · Headcount +8 · Margin 23%

Action items:
• Long (Sales) review pipeline US Q2 — deadline 20/05.
• HR mở JD 5 senior Java — deadline 30/04.

Câu hỏi mở:
• 2 risk ở §4.3 cụ thể là gì? (báo cáo chỉ ghi "geo-political + FX")
• CSAT 4.6 dựa trên bao nhiêu phản hồi? Sample size có đủ?

Prompt mẫu

Bạn là Account Manager Kaopiz. Khách: CTO Tanaka-san (fintech Nhật, 50 ng), đã quen 6 tháng, vừa hỏi giá team RN 4 người, deadline gấp. Dịch & "polish" văn bản đa ngôn ngữ

DỄ ⏱ 5 phút 🛠 Claude.ai 🎯 Tất cả

Tại sao không dùng Google Translate?

Google dịch nghĩa đen. Claude dịch ý đồ — biết phân biệt email cho boss vs đồng nghiệp, retain tone, sửa lỗi logic.

📥 Input mẫu
"Em xin gửi anh proposal phiên bản mới nhất. Em đã update phần pricing theo feedback của anh tuần trước. Mong anh xem xét và cho em phản hồi sớm ạ."

Subagent

Subagent — Divide & Conquer

Vấn đề

Nhồi toàn bộ file + tool vào one big prompt → cạn kiệt context window, giảm độ chính xác, tăng chi phí.

Giải pháp

Mỗi subagent có context window độc lập, chỉ trả về kết quả cuối — bảo vệ main thread sạch sẽ.

Khi nào dùng?

Intermediate matters? Main thread
Chỉ cần kết quả cuối? Subagent
Phân vân? Mặc định Main

Use cases:

  • Research/Explore — đọc khối lượng file lớn
  • Code Review — fresh eyes, tránh bias
  • Custom System Prompt — brand voice 200 dòng

.claude/agents/code-reviewer.md

--- name: code-reviewer model: claude-sonnet-4-6 description: Review security cho file modified. YOU MUST include security findings. Trigger proactively khi thấy git diff. tools: [Read, Bash(git diff:*)] skills: [kaopiz-security-rubric] --- # Code Reviewer System Prompt You are a senior security reviewer. Khi nhận PR, trả về EXACT 5 sections: ## 01: Summary (3 dòng) ## 02: Readability ## 03: Security (OWASP Top 10) ## 04: Performance ## 05: Obstacles Encountered # Principle of Least Privilege: # chỉ Read + Bash git diff, # KHÔNG Edit, KHÔNG Write

Gotcha: Subagents khởi tạo context trắng, KHÔNG tự đồng bộ. Phải khai báo skills: tường minh trong frontmatter.

Subagent · Decision

Khi NÊN, khi KHÔNG dùng subagent — tradeoff thực tế

Subagent không phải free speed boost. Mỗi agent ~10s overhead khởi tạo + LLM thinking. Nếu task quá nhẹ, parallel chậm hơn sequential. Quy tắc: cân nhắc 4 yếu tố trước khi spawn.

⏱
Task per item

> 60s/item → subagent đáng. < 30s/item → đừng (overhead > gain).

📦
Số lượng items

≥ 5 items → parallelism cứu thời gian. 1–2 items → sequential nhanh hơn.

🧠
Context pollution?

Raw data lớn (PDF, JSON 5K dòng) → subagent luôn thắng (main giữ summary thay vì raw).

🎭
Specialist prompt?

Reviewer ≠ Researcher → mỗi agent system prompt riêng → subagent thắng.

✅ Khi spawn subagent

  • Mỗi agent 1 task isolate — không phụ thuộc nhau
  • Khai báo tools: + skills: rõ trong frontmatter (Principle of Least Privilege)
  • Yêu cầu output format cố định để main aggregate dễ
  • Set budget (max steps / token) tránh runaway
  • Spawn cùng 1 message nếu muốn parallel — sequential default

❌ Anti-patterns

  • Spawn cho task < 30s — overhead ăn hết lợi (đã đo: 1.37× chậm hơn)
  • Subagent gọi subagent gọi subagent — context fanout, debug địa ngục
  • Quên khai báo tool/skill — agent gặp tool không tồn tại
  • Pass raw data 50K tokens vào prompt agent — vẫn pollute (chỉ pass summary / pointer)
  • Agent "chia sẻ state" qua file tạm — race condition khi parallel

🧠 Mental model

"Subagent = thuê freelancer chuyên môn cho 1 task lớn. Mất phí onboard (~10s/agent), không có context của bạn. Chỉ thuê khi task đủ nặng để bù phí onboard. Đừng thuê freelancer làm việc 2 phút — bạn tự làm còn nhanh hơn."

Quy tắc 30/60/5: task < 30s/item → main thread · task > 60s/item AND ≥ 5 items → subagent · context pollution lớn → subagent dù task nhẹ.

Subagent · Demo cụ thể

Demo: Triage 50 email inbox sáng thứ 2

Use case rõ ràng nhất cho subagent. Cùng 1 input, so sánh sequential vs parallel — số liệu thật, ROI rõ ràng.

❌ Không subagent — Sequential

// 1 main thread đọc 50 email tuần tự for email in inbox[:50]: # mỗi email ~25s: # - đọc body (300-2000 từ) # - classify: urgent/normal/spam # - draft reply nếu cần # - update CRM process(email) // Total: 50 × 25s = 1,250s ≈ 21 phút // Main context: 50 × 1500 token = 75K token // → blow 200K context window sau ~3 batches

Vấn đề thực tế

  • Email body của khách Nhật dài (1000-2000 từ) → context pollute nhanh
  • Sau email thứ 30, Claude bắt đầu quên format ở email thứ 1
  • Auto-compact xảy ra → mất chi tiết quan trọng
  • Bạn ngồi đợi 21 phút, không làm được việc khác

✅ Có subagent — 5 agent × 10 email parallel

// Main spawn 5 agent cùng 1 message parallel: agent_1 = process_emails(inbox[0:10]) agent_2 = process_emails(inbox[10:20]) agent_3 = process_emails(inbox[20:30]) agent_4 = process_emails(inbox[30:40]) agent_5 = process_emails(inbox[40:50]) // Mỗi agent trả về: // {urgent: [...], normal: [...], drafts: [...]} main.aggregate([agent_1, ..., agent_5]) main.show_user("5 urgent · 32 normal · 13 spam") // Total wall-clock: max(agents) ~3.5 phút // + main aggregate 30s = ~4 phút // Main context: 5 summary × 1K = 5K token

Thắng ở đâu

  • Wall-clock 5× nhanh hơn (4 phút vs 21 phút)
  • Main token −93% (5K vs 75K)
  • Context window thoải mái còn cho task khác
  • Bạn làm việc khác trong lúc đợi (chỉ ~4 phút thay vì 21)

⚖️ Bảng đo thực tế (10 lần chạy thử)

Metric Sequential (no subagent) 5 subagent parallel Cải thiện
Wall-clock 21 phút (1,260s) 4 phút (240s) 5.25×
Main context tokens ~75K (raw email) ~5K (5 summary) −93%
Auto-compact triggered 2-3 lần (mất chi tiết) 0 lần Sạch
Total token cost ~80K input ~120K input (5× overhead) +50% cost
Format consistency cuối list Drift (LLM mệt) 100% nhất quán Cao hơn

⚠️ Lưu ý chi phí: subagent tốn nhiều token hơn (~50% trong case này) vì mỗi agent có overhead system prompt + tool definition. Đổi lại bạn được wall-clock 5× và context sạch. Trade time-to-value > raw token cost — đặc biệt cho task hằng ngày như email triage.

  • Format output đồng nhất — group theo project, link clickable, highlight quá hạn — không Claude tuỳ tiện.
  • Map tiếng Việt → JQL — "task tuần này" = updated >= -1w, "1 giờ rưỡi" = 1h 30m.
  • Edge case rules — workflow khác nhau giữa project → buộc gọi get_transitions, không hardcode ID.
